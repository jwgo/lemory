# 🍋 Lemory 시작 가이드 — 따라오시면 돼요

10분이면 내 옵시디언 볼트가 "물어보면 대답하는 지식창고"가 돼요.
**전 과정 무료예요.** 무료 API 키 1개면 되고, 카드 등록은 필요 없어요.

---

## 0. 준비물

| 필요한 것 | 확인 방법 |
|---|---|
| Python 3.10 이상 | 터미널에서 `python3 --version` |
| 옵시디언 볼트 (`.md` 파일이 든 폴더면 돼요) | 경로만 알면 돼요. 예: `~/Documents/MyVault` |
| Gemini 무료 API 키 (2분이면 발급, 아래에서 안내) | — |

> 옵시디언을 안 써도 괜찮아요. 마크다운 파일이 들어 있는 폴더라면 뭐든
> 볼트로 쓸 수 있어요.

## 1. 설치

```bash
pipx install "git+https://github.com/jwgo/lemory"
```

`pipx`가 없다면 pip으로 설치해도 돼요:

```bash
pip install "git+https://github.com/jwgo/lemory"
```

## 2. 무료 키 발급 — 2분, 카드 필요 없어요

1. https://aistudio.google.com 접속 → 구글 계정으로 로그인
2. 왼쪽 위 **Get API key** → **Create API key**
3. 만들어진 키를 복사해요 (`AIza...` 형태)

이게 전부예요. **결제 수단을 등록하지 않는 한 요금이 나갈 방법 자체가
없어요.** 무료 한도를 넘으면 과금되는 게 아니라 잠깐 기다리게 될 뿐이에요.
Lemory가 무료 한도에 맞춰서 알아서 속도를 조절해요.

## 3. 설정 — 명령 하나면 돼요

```bash
lemory up                       # 대화형: 볼트 위치를 묻고 색인까지
lemory up ~/Obsidian/MyVault    # 질문 없이: 모드 자동 감지 → 색인 → 대시보드
lemory up ~/Vault --key <KEY>   # Gemini 클라우드 모드
```

`up`이 알아서 최적 모드를 골라요. Gemini 키가 있으면 클라우드, llama.cpp가
깔려 있으면 e5-small-ko-v2 임베딩 + Gemma 4 온디바이스 답변, 둘 다 없으면
검색 전용 로컬이에요. 대화형으로 실행하면 온디바이스 답변을 위한
`pip install "lemory[llama]"`도 제안해줘요. 번호 메뉴도 마법사도 없어요.
끝나면 바로:

```bash
lemory ask "지난주 회의에서 뭐 결정했지?"
```

답변 모델이나 임베더는 나중에 대시보드의 **설정 › 모델** 카드에서 바꿀 수
있어요.

## 4. 쓰는 법 — 한 번 설정하면, 그다음엔 그냥 쓰면 돼요

구조는 간단해요. **Lemory는 항상 떠 있는 로컬 백엔드**이고, 나머지 전부
(옵시디언 플러그인, Claude/MCP, 웹 UI, CLI)가 거기에 붙어요.

**① 한 번만 (설정)**
```bash
lemory up ~/Obsidian/MyVault      # 설정 + 첫 색인 + 서버까지 한 번에
```
키가 없어도 로컬 임베딩으로 바로 검색돼요. 답변(ask)까지 원하면 llama.cpp가
있을 때 **온디바이스 Gemma 4**(키 불필요)가 기본으로 잡히고, `--key <KEY>`를
주면 Gemini를 써요.

**② 계속 켜두기 (추천 — 상시 백엔드)**
```bash
lemory serve                       # http://127.0.0.1:8377
```
- 브라우저에서 웹 콘솔이 열려요. **현황**(노트/청크/그래프, 원클릭 재색인) ·
  **지식**(폴더 탐색, 백링크) · **검색**(출처 달린 답변) · **설정**. <kbd>⌘K</kbd>로
  어디서든 점프할 수 있어요.
- **켜둔 동안 노트를 고치면 몇 초 안에 자동으로 다시 색인돼요.** 수동 색인이
  필요 없어요.
- **언제 켜둬야 하나요?** 옵시디언 플러그인, 웹 대시보드, REST를 쓸 때요.
  로그인 시 자동 시작으로 두면 "항상 있는 기억"이 돼요.

**③ 가끔 한 줄 (서버 없이도 돼요)**
```bash
lemory ask "요새 하던 그거 어디까지 했지?"   # 터미널에서 바로
lemory search "결제 정책" · lemory recent · lemory remember "..."
```
이 명령들은 인덱스를 직접 열어서 동작해요. `serve`가 꺼져 있어도 돼요.

**④ 다시 해야 할 때**
- serve를 안 켜둔 채 노트를 많이 바꿨다면 → `lemory index` (변경분만, 빨라요)
- **임베딩 모델이나 프로바이더를 바꿨다면** → `lemory index` (벡터 공간이
  달라져서 전체를 다시 색인해요)
- 볼트 경로가 바뀌었거나 새 컴퓨터라면 → `lemory up`을 다시

**Claude/Cursor에 붙이기** — MCP는 `serve` 없이도 자체 엔진으로 떠요:
```bash
claude mcp add lemory -- lemory mcp --vault ~/Obsidian/MyVault --client claude-code
```

막히면 `lemory doctor`를 치세요. 지금 상태가 어떤지, 뭘 하면 되는지
알려줘요.

## 얼마나 무료인가요?

Gemini 무료 티어 기준으로 Lemory가 실제로 쓰는 양은 이래요:

| 작업 | API 소비 | 무료 한도로 가능한 양 |
|---|---|---|
| 최초 색인 | 청크 64개당 호출 1회 | 노트 1,000개(청크 약 3,000개) ≈ 47회 — **하루 한도의 5%** |
| 노트 수정 후 재색인 | 바뀐 노트의 청크만 | 사실상 0 |
| 볼트 전체 재색인 | **0회** (내용이 같으면 캐시가 적중해요) | 무제한 |
| 검색만 | 질의 임베딩 1회 (같은 질의는 캐시) | 하루 수백 회 |
| 질문 (`ask`) | 검색 1회 + 답변 생성 1회 | **하루 약 250회** |

핵심은 **임베딩 캐시**예요. 한 번 임베딩한 문단은 다시는 비용을 내지
않아요. 노트를 고쳤다가 되돌려도, 전체 재색인을 눌러도 0원이에요.

### 키 없이 100% 오프라인 — 전부 온디바이스, 데몬 없이

모든 로컬 경로가 **llama.cpp 엔진 하나에서** 돌아요. Ollama도, 관리할
서버도 없어요. GPU가 있으면 쓰고(Mac Metal, CUDA/Vulkan), 없으면 CPU로
돌아요. `lemory up`이 환경에 맞는 걸 알아서 골라요.

**모드 1 — ⭐ 최고 로컬 (추천): 질문·답변까지 전부 오프라인**

전체 스택이 온디바이스, 키 0개예요. llama.cpp가 있으면 `lemory up`이
기본으로 설정해요. 모델 파일 3개는 처음 쓸 때 자동으로 받아요.

- 임베딩: **e5-small-ko-v2** (한국어 특화, 384차원, 컴파일 불필요).
  KorMapleQA 하이브리드 **doc@8 0.899** — 측정해본 로컬 임베더 중 가장
  강했어요. 1024차원 Harrier(0.853)보다도 높아요.
- 답변: **Gemma 4 E4B**를 llama.cpp로 스트리밍해요. 웹 콘솔에서 더 가벼운
  **E2B**로 바꿀 수 있어요.
- 전용 리랭커도 있지만(`reranker = true`) **기본은 꺼져 있어요.** 강한
  임베더 위에서는 오히려 나빠진다는 게 측정 결과예요(아래 참고). 볼트
  내용은 **한 바이트도** 밖으로 안 나가요.

**모드 2 — 경량 로컬 (검색 전용)**

- 같은 e5-small-ko-v2 임베더를 답변 모델 없이 써요.
  `pip install "lemory[local]"`, 순수 파이썬이라 컴파일이 없어요.
  라즈베리파이급에서도 검색이 돼요. `ask`만 빠져요.

`local_embed_backend = auto`는 e5-small-ko-v2(측정상 최강)를 써요. 1024차원
Harrier를 원하면 `llamacpp`로 바꾸세요.

**최소 사양**

| 모드 | RAM | 디스크 | 참고 |
|---|---|---|---|
| 1 최고 로컬 (e5 + Gemma 4 E4B) | **8GB+ (16GB 권장)** | 약 4.5GB | CPU만으로도 돌아요. 8GB면 E2B 추천 |
| 2 경량 로컬 (검색 전용) | 4GB | 약 250MB | 라즈베리파이급도 가능해요 |
| 3 Gemini API | 무관 | 거의 0 | 사양 무관, 인터넷만 있으면 돼요 |

섞어 써도 돼요. **임베딩은 로컬로 두고 답변 생성만 API로** — 볼트 전체가
아니라 검색된 청크 몇 개만 밖으로 나가요.

> **재색인이 얼마나 걸릴지 미리 알려드려요.** `lemory index`는 실행 전에
> "노트 N개 · 임베딩 필요 청크 M개 · 예상 시간 X"를 보여줘요. 콘솔의 전체
> 재색인 버튼도 같은 예상치를 확인창으로 띄워요. 처음엔 기본값으로
> 추정하고, 한 번 색인한 뒤로는 **내 컴퓨터에서 실측된 속도**로 보정돼요.

> **⚠ 임베딩 모델을 바꾸면 전체 재색인이 일어나요.** 모델이 다르면 벡터
> 공간이 달라서 기존 색인과 비교할 수 없기 때문이에요. Lemory가 모델
> 변경을 감지해서 다음 색인 때 알아서 전체를 다시 임베딩해요. 큰 볼트라면
> 시간이나 쿼터가 한 번 더 들어요. 캐시는 모델별로 따로 보관되니까 예전
> 모델로 **되돌아가는 건 공짜예요.**

## 이런 걸 물어볼 수 있어요

볼트에 적어둔 것이라면 뭐든지요.

**업무·회의**
```bash
lemory ask "3분기 킥오프에서 예산 얼마로 잡았지?"
lemory ask "데이터플랫폼팀 리드가 누구고, 무슨 일 하는 팀이지?"
lemory ask "재택근무 정책, 작년이랑 지금이랑 뭐가 달라졌지?"   # 시점 비교
```

**공부한 것**
```bash
lemory ask "자바스크립트 이벤트 루프가 뭐였지? 내가 정리한 노트 기준으로"
lemory ask "사피엔스 읽으면서 밑줄 친 문장 뭐가 있었지?"
```

**게임·취미**
```bash
lemory ask "카오스 벨룸 가기 전에 준비물 뭐라고 적어놨더라?"
lemory ask "김치찌개 황금비율, 내 레시피 노트 기준으로"
```

**생활**
```bash
lemory ask "알러지 올라올 때 대처 순서 뭐였지?"
lemory ask "오사카에서 갔던 그 라멘집 이름이 뭐였지?"
lemory ask "전세 계약 갱신 거절당하면 뭐부터 해야 한다고 정리해놨지?"
```

**연결을 따라가는 질문 (멀티홉)** — Lemory가 특히 강한 영역이에요:
```bash
lemory ask "프로젝트 아틀라스 리드가 좋아하는 DB가 뭐더라?"
# 아틀라스 노트 → [[리드 이름]] 링크를 타고 → 그 사람 노트에서 답을 찾아와요
```

**시간이 섞인 질문** — "언제"를 이해해요:
```bash
lemory ask "요새 내가 읽던 책 뭐였지?"      # 넉 달 전 책이 아니라 지금 책
lemory ask "3월에 읽던 책은?"               # 과거를 물으면 과거가 나와요
lemory recent                               # 최근 만진 노트 (LLM 없이 바로)
```

**범위를 좁히는 질문** — 연산자로 범위를 자르고 검색해요:
```bash
lemory search "tag:회의록 예산 결정"         # 회의록 태그 안에서만
lemory search "folder:프로젝트 마감일"       # 특정 폴더 안에서만
lemory search "tag:일지"                     # 연산자만 쓰면 그 범위 최신순 목록
```

**기억을 남기기** — 읽기만 하는 창고가 아니에요:
```bash
lemory remember "회사 VPN 갱신은 매년 3월, 담당은 김하늘" --tags 업무
lemory context                               # 볼트 요약 한 번에 (에이전트에 넘기기 좋아요)
lemory import-chats conversations.json       # ChatGPT/Claude 대화 → 검색되는 노트
```

## LLM은 언제 켜나요? (어려운 멀티홉 질문)

기본값이 LLM 없이 도는 건 일부러 그런 거예요. 검색은 밀리초에 끝나고
비용은 0이고, 말 바꿔 묻기·한국어·키워드·오타 강건성에서 모델 없이도
이겨요. 대부분의 질문엔 이게 맞는 선택이에요.

언어 모델이 진짜 도움 되는 유일한 지점은 **아주 큰 볼트에서의 깊은
멀티홉**이에요. "A가 위치한 지역의 X"처럼 정답 노트가 링크로만 닿고
질문의 단어를 하나도 안 담은 경우요. 청크 4만 2천 개 나무위키 코퍼스에서
이건 모든 시스템에게 어려워요. 한국어 특화 e5 기본이 Lemory의 0-LLM 2-hop
full-support를 0.477까지 올렸고(옛 MiniLM 시절 0.14), qmd가 질의당 60초씩
LLM을 돌려도 그 0.333보다 앞서요. 그래도 미해결 과제로 BENCHMARKS에
남겨뒀어요.

Lemory도 같은 옵션 두 개를 제공하고, 기본은 꺼져 있어요:

```toml
# lemory.toml
query_expansion = true   # 검색 전에 질의를 여러 변형으로 다시 쓰기
rerank = true            # 융합 후 상위 후보를 LLM으로 다시 채점
```

호출별로도 돼요: `lemory search "..." --expand --rerank`.

**정직한 주의 두 가지. 둘 다 측정한 거예요.**

첫째, 모델이 아주 중요해요. 작은 로컬 모델로는 부족해요. 나무위키 2-hop
40문항 샘플, 범용 소형 모델(`qwen2.5:3b`) 기준:

| 설정 | 2-hop full-support | 지연 |
|---|---|---|
| 기본 (LLM 0회) | 0.125 | 20 ms |
| + query_expansion | 0.150 | 1.5초 |
| + rerank | 0.100 | 6.5초 |
| + expand + rerank | 0.075 | 7.3초 |

질의 확장은 조금 올랐고, **3B 모델의 rerank는 오히려 해가 됐어요**(잡음
섞인 점수로 맞는 청크를 강등시켜요). qmd가 2-hop 문서 커버리지 1.000을
찍는 건 목적 특화된 확장·리랭커 모델을 싣기 때문이에요. 그러니 빈손으로
돌아온 특정 멀티홉 질문에만 *좋은* 모델로 `query_expansion`을 써보고, 범용
`rerank`는 꺼두세요. 소형 로컬 모델이 깊은 멀티홉을 혼자 풀 거라는 기대는
안 하는 게 좋아요.

*전용* 질의확장 모델(qmd의 `qmd-query-expansion-1.7B`, MIT)은 2-hop 이득을
약 2배로 키워요(0.125→0.175 vs 범용 3B의 0.150). 특화가 효과는 있어요.
다만 영어/기술 질의로 학습돼서 짧은 한국어 질의에서 헛소리를 해요
("스우 테마곡"→"big data"). 한국어 볼트엔 나쁜 기본값이라, 써볼 가치는
있어도 고정할 가치는 없어요.

**전용 리랭커도 있지만 기본은 꺼져 있어요. 이유는 측정이에요.**
`reranker = true`면 상위 후보를 크로스인코더로 재정렬해요. 크로스인코더는
이미 검색된 것만 다시 줄 세우니까, 애초에 못 찾은 걸 찾아주진 못해요.
**강한 임베더 위에선 값어치가 거의 없어요** — KorMapleQA 전체(2,067문항),
e5-small-ko-v2 기본 위 실측:

| 리랭커 | doc@1 | doc@8 | 지연 |
|---|---|---|---|
| 없음 (기본) | 0.628 | 0.889 | 약 16ms/질의 |
| Qwen3-Reranker-0.6B | 0.605 | 0.892 | 약 1.9초/질의 |

<sub>이 A/B는 ANN 리콜 수정 전에 측정했어요. 결정의 근거인 방향(리랭커가
doc@1을 깎는다)은 그대로예요. 기본값의 현재 절대치는 BENCHMARKS §5e에
있어요 (doc@1 0.641 / doc@8 0.899).</sub>

Qwen3-Reranker는 90배 느려지면서 doc@1을 오히려 **떨어뜨렸어요**(이미 맞던
1위를 잘못 뒤집어요). 그래서 검색은 리랭커 없이 나가고, `reranker`는 맞는
노트가 결과엔 있는데 1위가 아닌 코퍼스를 위한 선택 옵션으로 남겨요.

### 로컬 스택, 티어별 정리

전부 빠르고 무료인 기본값 위에 얹는 선택지예요. 대시보드 **설정 › 모델**
카드나 `lemory.toml`에서 고르면 돼요.

1. **기본 임베더 (`lemory[local]`, e5-small-ko-v2)** — 한국어 특화, 순수
   파이썬, 384차원, 임베딩당 9ms, 컴파일 없음. KorMapleQA 하이브리드
   **doc@8 0.899**로 측정한 로컬 임베더 중 최강이에요.
2. **선택형 1024차원 임베더 (`lemory[llama]`, Harrier-0.6B)** — doc@8
   0.853으로 오히려 **낮았고** 더 무거워요(약 640MB, 질의당 100ms). 더 큰
   차원을 원하는 분을 위한 옵션으로만 남겼어요.
3. **정밀 모드 (전용 리랭커), 기본 꺼짐** — 강한 임베더 위에선 거의 도움이
   안 되거나 오히려 해가 돼요(위 표). 맞는 노트가 결과엔 있는데 1위가 아닌
   코퍼스에서만 켜세요.
4. **근거 있는 답변 (Gemma 4, 온디바이스)** — 같은 엔진이 Gemma 4 E4B를
   돌려서 `lemory ask`와 웹 콘솔이 완전 오프라인으로 답해요. 검색엔 필요
   없고 `ask`에만 써요.

둘째, LLM을 돌린다면 **무료 티어 API 말고 온디바이스로** 돌리세요. LLM
파이프라인이 느린 건 거의 항상 모델이 아니라 API 대기열이에요(측정 중에
저희도 `429` 벽에 부딪혔어요). M 시리즈 맥의 llama.cpp Gemma 4는 과금도
대기열도 없이 답해요. `lemory up`이 이걸 기본으로 설정해요.

## 큰 볼트도 돼요 — 어디까지?

- 나무위키 실제 문서 **1,469편(청크 약 42,000개, 위키링크 24,850개)** 을
  그대로 색인해서 벤치마크했어요. recall@8 1.00, 질의당 약 0.2초예요.
- 합성 스케일 테스트: 청크 5만 개에서 풀 검색 70ms.
- 색인은 증분식이라 볼트가 커져도 바뀐 노트만 다시 처리해요.
- 저장소는 SQLite 파일 하나예요. 청크 4만 개 볼트도 수백 MB 수준이에요.
- 아주 긴 문서도 괜찮아요. 제목·헤딩 구조를 따라 알아서 나눠요.

**현실적인 감각으로요.** 10년치 일기 + 회의록 + 회사 문서 전부 = 보통 노트
1만~5만 개, 청크 5만~30만 개예요. 그 위로도 가요. 청크 2만 개를 넘으면
**IVF-int8 벡터 인덱스로 자동 전환**돼요(여전히 numpy뿐, 벡터DB 없음).
실측: **청크 100만 개에서 질의당 5.9ms, 정확 검색 대비 recall@10 1.000,
RAM 2.9GB → 732MB** (BENCHMARKS §12b). 작은 볼트는 기존 방식 그대로라
손해가 없어요. 다만 청크 수십만 개의 최초 임베딩은 무료 티어로 며칠
걸리니, 큰 코퍼스는 유료 임베딩(1회 수천 원 수준)이 현실적이에요. PDF도
색인돼요 — `index_pdf = true` + `pip install 'lemory[pdf]'`.

## 뭔가 안 될 때

```bash
lemory doctor     # 볼트/키/API/인덱스 상태를 한 번에 진단
lemory status     # 색인 통계
```

| 증상 | 처방 |
|---|---|
| `No API key found` | `.env`나 환경변수에 `GEMINI_API_KEY=...` — `lemory up --key <KEY>`가 해줘요 |
| 429 / 잠깐 멈춤 | 무료 한도에 닿았어요. 알아서 기다렸다 재시도해요. 과금 아니에요 |
| 검색 결과가 이상해요 | `lemory index --full` 후 다시 해보세요. 그래도 이상하면 이슈 남겨주세요 |
| 콘솔이 안 열려요 | `lemory serve`가 켜져 있는지, 포트 8377이 겹치지 않는지 확인하세요 |
