# 🍋 Lemory 시작 가이드 — 따라만 하면 됩니다

10분이면 내 옵시디언 볼트가 "질문하면 대답하는 지식베이스"가 됩니다.
**전 과정 무료**입니다 (무료 API 키 1개, 카드 등록 없음).

---

## 0. 준비물

| 필요한 것 | 확인 방법 |
|---|---|
| Python 3.10 이상 | 터미널에서 `python3 --version` |
| 옵시디언 볼트 (그냥 `.md` 파일이 든 폴더면 됩니다) | 경로만 알면 됨, 예: `~/Documents/MyVault` |
| Gemini 무료 API 키 (2분 소요, 아래에서 발급) | — |

> 옵시디언을 안 써도 됩니다. 마크다운 파일이 들어 있는 폴더라면 무엇이든
> 볼트로 사용할 수 있습니다.

## 1. 설치

```bash
pipx install "git+https://github.com/jwgo/lemory"
```

`pipx`가 없다면 (`pip install pipx` 후 다시 실행하거나) 그냥 pip으로:

```bash
pip install "git+https://github.com/jwgo/lemory"
```

## 2. 무료 키 발급 (2분, 카드 필요 없음)

1. https://aistudio.google.com 접속 → 구글 계정 로그인
2. 왼쪽 위 **Get API key** → **Create API key**
3. 생성된 키 복사 (`AIza...` 형태)

이게 전부입니다. **결제 수단을 등록하지 않는 한 요금이 나갈 방법 자체가
없습니다** — 무료 한도를 넘으면 과금되는 게 아니라 그냥 잠깐 기다리게 됩니다.
Lemory는 무료 한도에 맞춰 스스로 속도를 조절합니다.

## 3. 설정 — 명령 하나면 충분합니다

```bash
lemory up                       # 대화형: 볼트 경로를 묻고 색인까지
lemory up ~/Obsidian/MyVault    # 질문 0개: 모드 자동 감지 → 색인 → 대시보드
lemory up ~/Vault --key <KEY>   # Gemini 클라우드 모드
```

`up`이 최적 모드를 자동 감지합니다 — Gemini 키가 있으면 클라우드, llama.cpp가
깔려 있으면 e5-small-ko-v2 임베딩 + Gemma 4 온디바이스 답변, 둘 다 없으면 검색
전용 로컬. 대화형으로 실행하면 온디바이스 Gemma 답변을 위해
`pip install "lemory[llama]"`를 제안합니다. 번호 메뉴도 마법사도 없습니다. 끝나면 바로:

```bash
lemory ask "지난주 회의에서 뭐 결정했지?"
```

답변 모델이나 임베더(Gemma E4B/E2B, 임베더 백엔드, 리랭커)는 나중에 대시보드의
**설정 › 모델(Models)** 카드에서 바꿀 수 있습니다.

## 4. 쓰는 법 — 한 번 설정, 그다음엔 계속

머릿속 모델은 간단합니다: **Lemory는 상시 떠 있는 로컬 백엔드**이고, 나머지 전부
(옵시디언 플러그인, Claude/MCP, 웹 UI, CLI)가 거기에 붙습니다.

**① 한 번만 (설정)**
```bash
lemory up ~/Obsidian/MyVault      # 설정 + 첫 색인 + 서버까지 한 번에
```
키가 없어도 로컬 임베딩으로 바로 검색됩니다. 답변(ask)까지 원하면 `lemory up`이
llama.cpp가 깔려 있을 때 **온디바이스 Gemma 4**(키 불필요)를 기본으로 설정하고,
`--key <KEY>`를 주면 Gemini를 씁니다.

**② 계속 켜두기 (권장 — 상시 백엔드)**
```bash
lemory serve                       # http://127.0.0.1:8377
```
- 브라우저에서 웹 콘솔: **현황**(노트/청크/그래프·원클릭 재색인) · **지식**(폴더
  트리 탐색·백링크) · **검색**(출처 달린 답변, 검색만 하면 ms 즉답) · **설정**
  (임베딩 모델·검색 동작을 화면에서) · <kbd>⌘K</kbd> 어디서든 점프.
- **켜둔 동안 볼트를 편집하면 몇 초 안에 자동 재색인** — 수동 색인 불필요.
- **언제 켜둬야 하나:** 옵시디언 플러그인, 웹 대시보드, REST/자체 에이전트를 쓸 때는
  serve가 떠 있어야 합니다. 로그인 시 자동 시작으로 두면 "항상 있는 기억"이 됩니다.

**③ 가끔 한 줄 (서버 없이도 됨)**
```bash
lemory ask "요새 하던 그거 어디까지 했지?"   # 터미널에서 바로
lemory search "결제 정책" · lemory recent · lemory remember "..."
```
이 CLI들은 인덱스를 직접 열어 동작하므로 `serve`가 꺼져 있어도 됩니다.

**④ 다시 해야 할 때**
- serve를 안 켜둔 채 노트를 많이 바꿨다 → `lemory index` (변경분만, 빠름)
- **임베딩 모델/프로바이더를 바꿨다** → `lemory index` (벡터 공간이 달라 전체 재색인)
- 볼트 경로가 바뀌었거나 새 PC → `lemory up` 다시

**Claude/Cursor 등에 붙이기** — MCP는 `serve` 없이도 자체 엔진으로 뜹니다:
```bash
claude mcp add lemory -- lemory mcp --vault ~/Obsidian/MyVault --client claude-code
```

막히면 언제든 `lemory doctor` — 지금 어느 티어인지, ask가 되는지, 뭘 하면 되는지
알려줍니다.

## 얼마나 무료인가요?

Gemini 무료 티어 기준, Lemory의 실제 소비량은 이렇습니다:

| 작업 | API 소비 | 무료 한도로 가능한 양 |
|---|---|---|
| 최초 인덱싱 | 청크 64개당 임베딩 호출 1회 | 노트 1,000개(≈청크 3,000개) ≈ 호출 47회 — **하루 한도의 5%** |
| 노트 수정 후 재색인 | 바뀐 노트의 청크만 | 사실상 0 |
| 볼트 전체 재색인 | **0회** (내용이 같으면 캐시 적중) | 무제한 |
| 검색 (검색만) | 질의 임베딩 1회 (같은 질의는 캐시) | 하루 수백 회 |
| 질문 (`ask`) | 검색 1회 + 답변 생성 1회 | **하루 약 250회 질문** |

핵심은 **콘텐츠 해시 임베딩 캐시**입니다: 한 번 임베딩한 문단은 다시는 비용을
내지 않습니다. 노트를 고쳤다가 되돌려도, 전체 재색인을 눌러도 0원입니다.

### 키 없이 100% 오프라인 — 전부 온디바이스, 데몬 없음

모든 로컬 경로가 **하나의 llama.cpp 엔진에서** 돕니다 — Ollama도, 관리할 서버도
없고, GPU가 있으면 어디서든(Mac Metal, Linux/Windows CUDA/Vulkan, 없으면 CPU
오프로드) 씁니다. `lemory up`이 당신 환경에 맞는 걸 자동으로 고릅니다:

**모드 1 — ⭐ 최고 로컬 (추천): 질문·답변까지 전부 오프라인**

전 스택 온디바이스, 키 0 — llama.cpp가 있으면 `lemory up`이 기본으로 설정합니다.
답변 모델용 `pip install "lemory[llama]"`를 제안하고, GGUF 3개는 첫 사용 때 자동 다운로드됩니다.

- 임베딩: **e5-small-ko-v2** (dragonkue 한국어 특화 multilingual-e5-small, fastembed
  384차원, 무컴파일). KorMapleQA 하이브리드 **doc@8 0.889** — 측정한 로컬 임베더 중
  최강, 1024차원 Harrier(0.853)보다 높음.
- 답변: **Gemma 4 E4B** (Q4_K_M GGUF)를 llama.cpp GPU로 스트리밍. 웹 콘솔에서 더
  가벼운 **E2B**로 전환 가능. `lemory[llama]`는 이 답변 모델용이에요.
- 전용 리랭커는 있지만(`reranker = true`) **기본 꺼짐** — 강한 임베더 위에선 소형
  리랭커가 측정상 오히려 나빴어요(아래). 볼트 내용은 밖으로 **한 바이트도** 안 나감.

**모드 2 — 경량 로컬 (검색 전용): 답변 모델 없이**

- 같은 **e5-small-ko-v2** 임베더를 Gemma 답변 없이 — `pip install "lemory[local]"`,
  순수 파이썬 ONNX, 무컴파일, 최소 풋프린트. 라즈베리파이급에서도 검색+시맨틱
  임베딩, `ask`는 제외.

`local_embed_backend = auto`는 e5-small-ko-v2(측정상 최강)를 씁니다. 1024차원 Harrier를
원하면 `llamacpp`로.
임베딩만으로 `ask`를 빼고 전부 동작하며, `ask`는 생성기(최고 로컬의 온디바이스
Gemma 4, 또는 Gemini 키)가 필요합니다.

**최소 사양**

| 모드 | RAM | 디스크 | 비고 |
|---|---|---|---|
| 1 최고 로컬 (e5 + Gemma 4 E4B) | **8GB+ (권장 16GB)** | ~4.5GB | CPU만으로 동작, Metal/GPU면 답변이 시원. 8GB면 E2B로 |
| 2 경량 로컬 (검색 전용) | 4GB | ~250MB | 라즈베리파이급도 가능 |
| 3 Gemini API | 아무거나 | ~0 | 컴퓨터 사양 무관, 인터넷 필요 |

혼합도 됩니다: **임베딩은 로컬 유지, 답변 생성만 API** — 볼트 전체가 아니라
검색된 청크 몇 개만 밖으로 나갑니다.

> **재색인이 얼마나 걸릴지는 미리 알려줍니다.** `lemory index`는 실행 전에
> "노트 N개 · 임베딩 필요 청크 M개 · 예상 시간 X"를 출력하고, 콘솔의 전체
> 재색인 버튼도 같은 예상치를 확인창으로 보여줍니다. 예상 속도는 처음엔
> 프로바이더 기본값, 한 번 색인한 뒤로는 **당신 환경에서 실측된 속도**로
> 보정됩니다.

> **⚠ 임베딩 모델을 바꾸면 전체 재색인이 일어납니다.** 모델이 다르면 벡터
> 공간이 달라서 기존 색인과 비교 자체가 무의미하기 때문입니다. Lemory가 모델
> 변경을 자동 감지해 다음 색인 때 볼트 전체를 새 모델로 다시 임베딩합니다 —
> 큰 볼트라면 시간(로컬 모델) 또는 쿼터(API)가 한 번 더 듭니다. 캐시는 모델별로
> 따로 보관되므로 예전 모델로 **되돌아가는 것은 공짜**입니다.

## 이런 걸 물어볼 수 있습니다

볼트에 적어둔 것이라면 무엇이든. 실제 사용 패턴별 예시:

**업무·회의**
```bash
lemory ask "3분기 킥오프에서 예산 얼마로 잡았지?"
lemory ask "데이터플랫폼팀 리드가 누구고, 무슨 일 하는 팀이지?"
lemory ask "재택근무 정책, 작년이랑 지금이랑 뭐가 달라졌지?"   # 시점 비교
```

**공부한 것**
```bash
lemory ask "자바스크립트 이벤트 루프가 뭐였지? 내가 정리한 노트 기준으로"
lemory ask "사피엔스 읽으면서 밑줄 친 문장 뭐가 있었지?"
```

**게임·취미**
```bash
lemory ask "카오스 벨룸 가기 전에 준비물 뭐라고 적어놨더라?"
lemory ask "김치찌개 황금비율, 내 레시피 노트 기준으로"
```

**생활**
```bash
lemory ask "알러지 올라올 때 대처 순서 뭐였지?"
lemory ask "오사카에서 갔던 그 라멘집 이름이 뭐였지?"
lemory ask "전세 계약 갱신 거절당하면 뭐부터 해야 한다고 정리해놨지?"
```

**연결을 따라가는 질문 (멀티홉)** — Lemory가 특히 강한 영역:
```bash
lemory ask "프로젝트 아틀라스 리드가 좋아하는 DB가 뭐더라?"
# 아틀라스 노트 → [[리드 이름]] 링크를 타고 → 그 사람 노트에서 답을 찾아옵니다
```

**시간이 섞인 질문** — "언제"를 이해합니다:
```bash
lemory ask "요새 내가 읽던 책 뭐였지?"      # 넉 달 전 책이 아니라 지금 책
lemory ask "3월에 읽던 책은?"               # 과거를 물으면 과거가 나옴
lemory recent                               # 최근 만진 노트 (LLM 없이 즉시)
```

**범위를 좁히는 질문** — 연산자로 스코프를 자르고 검색합니다:
```bash
lemory search "tag:회의록 예산 결정"         # 회의록 태그 안에서만
lemory search "folder:프로젝트 마감일"       # 특정 폴더 안에서만
lemory search "tag:일지"                     # 연산자만 쓰면 = 그 범위 최신순 목록
```

**기억을 쓰기** — 읽기만 하는 지식창고가 아닙니다:
```bash
lemory remember "회사 VPN 갱신은 매년 3월, 담당은 김하늘" --tags 업무
lemory context                               # 볼트 요약 한 방 (에이전트에 파이프)
lemory import-chats conversations.json       # ChatGPT/Claude 대화 → 검색되는 노트
```

## LLM은 언제 켜나요 (어려운 멀티홉 질문)

기본값이 LLM 없이 도는 건 의도된 설계입니다. 검색은 밀리초, 비용은 0이고,
벤치마크상 패러프레이즈·한국어·키워드·오타 강건성에서 모델 없이도 이깁니다.
압도적 다수의 질문엔 이게 맞는 트레이드입니다.

언어 모델이 진짜로 도움 되는 유일한 지점은 **거대 볼트에서의 깊은 멀티홉**입니다.
"A가 위치한 지역의 X" 처럼 정답 노트가 링크로만 닿고 질문의 단어를 전혀
포함하지 않는 경우죠. ~42k 청크 나무위키 코퍼스에서 이건 모든 시스템에 어렵습니다.
Lemory의 0-LLM 2-hop full-support는 ~0.14이고, 정직한 이유는 정답 노트 자체가
어휘·의미 신호를 거의 안 갖기 때문입니다(BENCHMARKS 5e에 상시 미해결 과제로
기록). 이걸 푸는 시스템은 LLM으로 풉니다: qmd의 풀 로컬-LLM 파이프라인은 같은
코퍼스에서 2-hop 문서 커버리지 1.000에 도달합니다. 다만 질의당 ~60초.

Lemory도 같은 노브 두 개를 제공하고, 기본은 꺼져 있습니다:

```toml
# lemory.toml
query_expansion = true   # 검색 전에 질의를 변형들로 재작성
rerank = true            # 융합 후 상위 후보를 LLM으로 재채점
```

또는 호출별로: `lemory search "..." --expand --rerank`.

**정직한 주의 두 가지, 둘 다 실측.**

첫째, 모델이 엄청나게 중요하고 작은 로컬 모델로는 부족합니다. 나무위키
코퍼스의 2-hop 40문항 샘플, 범용 소형 로컬 모델(`qwen2.5:3b`) 기준:

| 설정 | 2-hop full-support | 지연 |
|---|---|---|
| 베이스라인 (0 LLM) | 0.125 | 20 ms |
| + query_expansion | 0.150 | 1.5 s |
| + rerank | 0.100 | 6.5 s |
| + expand + rerank | 0.075 | 7.3 s |

질의 확장은 소폭 상승, **3B 모델의 rerank는 오히려 해가 됐습니다**(잡음 섞인
관련도 점수로 맞는 청크를 강등). qmd가 2-hop 1.000을 찍는 건 범용 소형 LLM이
아니라 목적 특화된 확장·리랭커 모델을 싣기 때문입니다. 그러니 빈손으로 돌아온
특정 멀티홉 질문에 한해 *유능한* 모델로 `query_expansion`을 시도하고, 범용
`rerank`는 꺼 두고, 소형 로컬 모델이 깊은 멀티홉을 혼자 풀 거란 기대는 마세요.

*전용* 질의확장 모델(실측: qmd 자체 `qmd-query-expansion-1.7B`, MIT)은 2-hop에서
범용 대비 이득을 약 2배로 키웁니다(0.125→0.175 vs 범용 3B의 0.150, 특화가
효과 있음). 다만 영어/기술 질의 분포로 학습돼 짧은 한국어 질의에서 환각합니다
("스우 테마곡"→"big data"). 한국어 우선 볼트엔 나쁜 기본값이라, 시도할 가치는
있어도 고정할 가치는 없습니다.

**전용 리랭커는 있지만 기본 꺼짐 — 이유는 측정.** `reranker = true`면 상위 후보를
융합 대신 크로스인코더로 재정렬합니다. 크로스인코더는 이미 검색된 것만 재정렬해서
doc@1은 올릴 수 있어도 깊은 멀티홉의 리콜 실패는 못 고쳐요. **강한 임베더 위에선
값어치가 거의 없습니다** — KorMapleQA v2 전체(2067), e5-small-ko-v2 기본 위 실측:

| 리랭커 | doc@1 | doc@8 | 지연 |
|---|---|---|---|
| 없음 (기본) | 0.628 | 0.889 | ~20ms/q |
| Qwen3-Reranker-0.6B | 0.605 | 0.892 | ~1.9s/q |

Qwen3-Reranker는 오히려 doc@1을 **떨어뜨렸어요**(0.6B 리랭커가 이미 맞는 top을
잘못 뒤집음) — 90배 느린 대가로. (예전 fastembed jina-reranker-v2 경로는 이제
제거됐는데, ~40배 지연에 doc@8 ~+1pt였습니다.) 그래서 검색은 **리랭커 없이**
출시하고(임베더+BM25+링크그래프 융합이 이미 잘 랭킹함), `reranker`는 맞는 노트가
결과엔 있는데 1위가 아닌 코퍼스용 opt-in 정밀 노브로 남깁니다.

### 로컬 스택, 티어별

위의 모든 건 빠르고 무료인 기본값 위에 얹는 선택지입니다. 대시보드의
**설정 › 모델(Models)** 카드나 `lemory.toml`에서 원하는 티어를 고르세요:

1. **기본 임베더 (`lemory[local]`, e5-small-ko-v2):** dragonkue 한국어 특화
   multilingual-e5-small을 fastembed(순수 파이썬 ONNX, 384차원, ~9ms/임베딩,
   무컴파일)로. 전체 KorMapleQA v2 하이브리드 **doc@8 0.889** — 측정한 로컬 임베더
   중 최강, 1024차원 Harrier(0.853)·구 MiniLM(0.788)보다 높고, 테스트한 영어·초장문
   코퍼스에서도 Harrier한테 안 짐. `local_embed_backend = auto`가 이걸 씁니다.
2. **선택형 1024차원 임베더 (`lemory[llama]`, Harrier-OSS-0.6B):** 인프로세스
   llama.cpp(Metal/GPU), doc@8 0.853. 여기선 e5-small-ko-v2보다 **낮게** 나왔고
   더 무거워요(~640MB, ~100ms/질의 vs ~9ms) — 더 큰 차원을 원하는 사람을 위해
   옵션(`local_embed_backend = "llamacpp"`)으로 남겼지만 기본은 아닙니다.
3. **정밀 모드 (+ 전용 리랭커), 기본 꺼짐:** `reranker = true`면 상위 후보를
   크로스인코더로 재정렬합니다. 강한 로컬 임베더 위에선 거의 도움이 안 되거나
   오히려 해가 되고(위 표) 지연이 큽니다 — 그래서 **기본 OFF**. 맞는 노트가
   결과엔 있는데 1위가 아닌 코퍼스에서만 켜세요.
4. **근거 있는 답변 (+ Gemma 4, 온디바이스):** 같은 `lemory[llama]` 엔진이
   **Gemma 4 E4B**(Q4_K_M GGUF)를 굴려 `lemory ask`와 웹 콘솔이 완전 오프라인으로
   답합니다. 콘솔에서 더 가벼운 **E2B**로 전환 가능. 검색엔 전혀 필요 없고 `ask`만 씀.

둘째, LLM을 돌릴 때는 **무료 티어 API가 아니라 온디바이스로** 돌리세요. LLM
파이프라인의 느림은 거의 항상 모델이 아니라 API 큐입니다(측정 중 바로 그
`429 credits depleted` 벽에 부딪혔습니다). M 시리즈 노트북의 llama.cpp Gemma 4는
질의당 과금·큐 없이 답합니다. `lemory up`이 이걸 기본 온디바이스로 설정합니다.

## 큰 볼트도 됩니다 — 어디까지?

- 나무위키 실문서 **1,469편 (청크 33,375개, 위키링크 24,850개)** 를 그대로
  색인해 벤치마크했습니다 — 검색 품질 recall@8 1.00, 질의당 약 0.2초.
- 합성 스케일 테스트: 청크 5만 개에서 풀 하이브리드+그래프 검색 70ms.
- 색인은 증분식이라 볼트가 커져도 "바뀐 노트만" 다시 처리합니다.
- 저장소는 SQLite 파일 하나 — ~42k 청크 볼트도 수백 MB 수준입니다.
- 아주 긴 단일 문서도 괜찮습니다 — 제목·헤딩 구조를 따라 자동 분할됩니다.

**현실적인 감각으로**: 10년치 일기 + 회의록 + 회사 문서 전부 = 보통 노트
1만~5만 개, 청크 5만~30만 개 규모입니다. 그리고 그 위로도 갑니다 — 청크
2만 개를 넘으면 **IVF-int8 벡터 인덱스로 자동 전환**됩니다 (여전히 numpy뿐,
FAISS·벡터DB 없음). 실제 임베딩 분포로 실측: **청크 100만 개(≈ 노트 15만 개
볼트)에서 질의당 5.9ms, 정확 검색 대비 recall@10 = 1.000, RAM 2.9GB → 732MB**
(BENCHMARKS §12b). 작은 볼트는 기존 정확 검색 그대로라 손해가 없습니다.
다만 청크 수십만 개의 최초 임베딩은 무료 티어로는 며칠 걸리므로 큰 코퍼스는
유료 임베딩(1회 수천 원 수준)이 현실적입니다. PDF도 색인됩니다 —
`lemory.toml`에 `index_pdf = true` + `pip install 'lemory[pdf]'` (opt-in).

## 뭔가 안 될 때

```bash
lemory doctor     # 볼트/키/API/인덱스 상태를 한 번에 진단
lemory status     # 색인 통계
```

| 증상 | 처방 |
|---|---|
| `No API key found` | `.env` 또는 환경변수에 `GEMINI_API_KEY=...` — `lemory up --key <KEY>`가 해줍니다 |
| 429 / 잠깐 멈춤 | 무료 한도 도달 — Lemory가 알아서 기다렸다 재시도합니다. 과금 아님 |
| 검색 결과가 이상함 | `lemory index --full` 후 다시. 그래도 이상하면 이슈 환영 |
| 콘솔이 안 열림 | `lemory serve`가 켜져 있는지, 포트 8377 충돌 확인 |
