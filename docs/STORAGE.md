# 왜 아직도 SQLite인가 — 대안 실측 리포트 (2026-07)

"SQLite 말고 대안은? 충분히 좋으면 바꿔도 됨"이라는 질문에 주장 대신 측정으로
답한 문서. 후보는 임베디드 세계의 실질적 전부: **DuckDB**(모던 분석형 임베디드
DB + FTS 확장), **LanceDB**(임베디드 벡터DB + tantivy 계열 FTS — cognee가 쓰는
그 물건), 그리고 참고로 sqlite-vec/Postgres 계열. 측정 스크립트는
`benchmarks/bench_storage_alternatives.py` / `bench_lancedb_vector.py`.

## Lemory의 실제 워크로드가 뭔지부터

스토리지 벤치마크는 워크로드를 잘못 잡으면 의미가 없다. Lemory가 저장소에
시키는 일은 분석 쿼리가 아니라 이것들이다:

| 코드 경로 | 저장소 연산 | 빈도 |
|---|---|---|
| 첫 색인 | 청크 대량 INSERT + FTS 색인 | 볼트당 1회 |
| 감시자 증분 동기화 | 노트 단위 DELETE+INSERT+FTS 갱신 | **분당 수 회** |
| 검색 lexical leg | BM25 top-48 | 질의당 1-2회 |
| ANN 재채점 | PK 26건 배치 조회 | 질의당 1회 |
| CLI ↔ 서버 동시 접근 | 두 프로세스가 같은 파일에 읽기+쓰기 | 상시 |

벡터 검색은 이미 SQLite **밖**에 있다(numpy 정확 스캔 / IVF-int8 사이드카,
BENCHMARKS §12b). 그러니 "벡터DB로 바꾸면 빨라진다"는 프레임은 시작부터
성립하지 않는다 — 바꿔서 얻을 수 있는 건 FTS와 트랜잭션 스토어뿐이다.

## 실측 결과 (50k 청크, 동일 코퍼스·동일 질의)

| | 첫 색인 (50k) | 증분 100노트 | FTS top-48 | PK 26건 조회 | 디스크 | 2프로세스 쓰기 |
|---|---|---|---|---|---|---|
| **SQLite+FTS5 (현재)** | **2.8s** | **0.16s** | 42.4ms | **0.11ms** | 231MB | ok |
| DuckDB+FTS | 196.3s | 13.18s | 308.7ms | 28.97ms | 179MB | ok* |
| LanceDB (native FTS) | 2.9s | 0.26s | **8.6ms** | 8.25ms | **23.9MB** | 미측정 |

\* DuckDB의 두 번째 프로세스 쓰기는 두 연결의 설정이 바이트 단위로 동일할 때만
성립한다 (설정이 다르면 ConnectionException). FTS 인덱스가 걸린 상태의 동시
쓰기 의미론은 별도 검증 필요.

읽는 법: DuckDB는 다섯 축 전부에서 탈락이다 — FTS 색인이 증분 갱신이 없어서
첫 색인 70배, 증분 동기화 82배 느리고, 질의도 7배, PK 조회는 263배 느리다.
분석(OLAP) 엔진에 OLTP+검색 워크로드를 시킨 결과이며 DuckDB가 나쁜 DB라는
뜻이 아니라 **이 워크로드용이 아니라는** 뜻이다.

LanceDB의 native FTS는 **정직하게 기록한다: 우리보다 5배 빠르다** (8.6ms vs
42.4ms — 흔한 단어투성이의 worst-case 코퍼스 기준). 대신 재채점·후보 조회가
지나는 PK 배치 경로가 75배 느리고(0.11→8.25ms, 질의당 2회 이상 지나감),
이 8.6ms의 이득은 하이브리드 총 지연(50k에서 ~48ms, LLM 답변 경로는 1–2s)
안에서 사용자가 느낄 수 없는 크기다.

## 벡터 쪽: LanceDB IVF-PQ vs 우리 IVF-int8 (200k × 768d, 실제 임베딩 매니폴드)

| | 인덱스 빌드 | 질의 | recall@10 (vs 정확 스캔) | 저장 |
|---|---|---|---|---|
| **Lemory IVF-int8 + f32 재채점** | **6.4s** | **2.6ms** | **1.000** | **146MB** |
| LanceDB IVF-PQ (nprobes=20) | 243.8s | 13.7ms | 0.528 | 607MB |
| LanceDB IVF-PQ (+refine 10) | 243.8s | 14.4ms | 0.953 | 607MB |

같은 코퍼스·같은 질의·같은 시드 (bench_scale.py의 실제 gemini-embedding
매니폴드). PQ의 낮은 기본 recall(0.528)은 refine으로 회복되지만 그래도
0.953에서 멈추고, 빌드는 38배·질의는 5.5배 느리다. 공정한 각주: LanceDB의
설계 목표는 RAM을 초과하는 수천만~수억 벡터의 디스크 기반 검색이고, 거기서는
LanceDB가 이긴다. 개인 지식베이스는 그 규모에 도달하지 않는다 (1M 청크 =
노트 15만 개 = int8 732MB RAM).

## 숫자 밖의 결정 요인 (이게 더 크다)

**1. DuckDB FTS는 증분 갱신이 없다.** 문서 하나를 고치면
`PRAGMA create_fts_index(..., overwrite=1)` — 인덱스 전체 재구축이다.
Lemory의 감시자는 노트 저장마다 동기화한다. 노트 하나 고칠 때마다 수 초짜리
전체 FTS 리빌드를 하는 아키텍처는 라이브 워크로드에서 실격이다. (DuckDB의
동시성 모델도 단일 읽기-쓰기 프로세스 — CLI를 서버 옆에서 못 쓴다.)

**2. LanceDB는 데이터를 조각 파일 디렉터리로 관리한다.** "볼트 옆의 SQLite
파일 하나, `cp` = 백업"이라는 운영 스토리가 사라지고, pyarrow(+수십 MB 네이티브
휠) 의존성이 붙는다. FTS 토크나이저는 영어 중심이라 한국어는 어차피 우리
바이그램 레이어를 다시 얹어야 한다 — 즉 한국어 품질은 그대로인데 의존성만 는다.

**3. SQLite는 표준 라이브러리다.** `pip install lemory`가 끌고 오는 네이티브
의존성 0개. FTS5는 25년째 단일 파일 OLTP + 증분 FTS라는 정확히 이 워크로드를
위해 존재해 왔고, WAL + busy_timeout으로 두 프로세스 동시 접근도 검증돼 있다
(위 표의 2nd-process-write 열).

**4. 우리가 이미 병목을 두 번 옮겼다.** 벡터 스캔(→IVF-int8, 1M 청크 5.9ms)과
FTS OR-질의(→2단계 AND-우선, 60→37ms)는 둘 다 SQLite를 유지한 채 해결됐다.
현재 남은 저장소 비용 중 엔진 교체로 사라지는 것이 없다.

## 결론

**바꾸지 않는다 — 데이터가 그렇게 말한다.** 교체 후보들이 이기는 축(분석 질의,
컬럼 스캔, 페타바이트급 벡터)은 Lemory가 쓰지 않는 축이고, Lemory가 매초 쓰는
축(증분 FTS, 노트 단위 upsert, PK 배치 조회, 2-프로세스 접근, 배포 마찰 0)에서는
SQLite가 이기거나 유일하게 가능하다. 재검토 트리거도 명시해 둔다: (a) FTS가
다시 병목이 되는 실측이 나오거나, (b) 단일 파일 요구를 버릴 만큼 멀티유저
요구가 생기면 — 그때는 tantivy(FTS)나 Lance(벡터)를 **부분** 교체 후보로 다시
측정한다.
