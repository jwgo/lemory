# memvid (github.com/memvid, ~16k★) — 하이프 검증 + 한국어 실측 (2026-07)

한국 AI 스레드에서 "LoCoMo +35% over SOTA / 처리량 1,372배 / P50 0.025ms"로
홍보되며 등장. 정찰 + 실측으로 냉정하게 봤다.

## 먼저 인정할 진짜 강점 (실측 대상 아님, 그대로 인정)

- **멀티모달**: `clip`(이미지 검색) + `whisper`(음성 전사)를 feature flag로.
  Lemory(텍스트/마크다운)에 없는 축 — 여기선 memvid가 앞선다.
- **폴리글랏 SDK**: Rust 코어 + Node/Python/CLI. Lemory는 Python+MCP.
  Rust·Node 에이전트에 메모리를 심으려면 memvid가 유통에서 유리.
- **단일 파일 `.mv2` + Rust 성능**: 인덱스 탐침 자체는 빠르다.

## 세 헤드라인 숫자는 마케팅이다 (작성자 본인 글로 반증)

| 주장 | 실제 |
|---|---|
| "P50 0.025ms" | 쿼리 임베딩을 뺀 **인메모리 인덱스 탐침 1스텝**. 작성자 본인이 v2 Show HN에서 "end-to-end <17ms", v1 시절엔 "~800-900ms"라고 씀. end-to-end 대비 **최대 680배 차이** |
| "처리량 1,372배" | **인프로세스 vs 네트워크 벡터DB**(Pinecone류) 비교. SQLite·FAISS 등 임베디드면 뭐든 나오는 배수 — memvid 고유 알고리즘 우위 아님 |
| "+35% over SOTA" (LoCoMo) | 실은 **경쟁작 평균 대비** +35%. 실제 최고 베이스라인(full-context 72.9%) 대비는 +17%. 그것도 자체 채점 + 논란 있는 LLM-judge 벤치(memvidbench도 "일부 벤더가 이의 제기"라고 명시) |

또한 v1의 시그니처였던 **"텍스트를 QR로 비디오에 인코딩"은 v2에서 폐기**됐다
(README: "v1 QR-based memory is deprecated"). 지금 `.mv2`는 그냥 커스텀
단일파일 임베디드 DB — "비디오 메모리" 브랜딩은 기술적 실체 없는 잔재다.

## 카테고리가 다르다

memvid = **에이전트 메모리 API/포맷**("RAG 스택을 대체"), 소스오브트루스는
불투명 바이너리 `.mv2`. Lemory = **사람이 쓴 옵시디언/마크다운 볼트** 백엔드,
소스오브트루스는 당신의 `.md` 파일. 스레드는 둘을 뭉뚱그렸다. 겹치는 축은
"검색 품질" 하나뿐이라 거기서만 실측했다.

## 실측: 한국어 검색 (같은 코퍼스, 양쪽 다 로컬·무API)

KorQuAD 실문단 113개 · 인간 작성 질문 120개 · 문단 recall@1
(`benchmarks/run_memvid_korean.py`):

| 시스템 (로컬 임베더) | 언어 | recall@1 | p50 지연 (end-to-end) |
|---|---|---|---|
| memvid v1 (all-MiniLM-L6-v2, EN) | 영어 대조군 | **0.933** | — |
| memvid v1 (all-MiniLM-L6-v2, EN) | **한국어** | **0.050** | 45.5 ms |
| **Lemory 키리스** (e5-small-ko-v2) | **한국어** | **0.958** | **17.0 ms** |

읽는 법: memvid은 영어에선 0.933으로 멀쩡하다 — **한국어에서만 랜덤(0.009)
수준으로 붕괴**한다. 영어 전용 임베더가 유일한 변수임이 대조군으로 격리됐다.
그리고 쿼리 임베딩 포함 end-to-end 지연은 memvid가 **더 느리다**(45.5 vs
17.0ms) — 홍보된 "0.025ms"와 정반대.

정직한 라벨: **테스트한 건 pip `memvid` v1**(비디오층 폐기 전 버전). v2(Rust)는
여기서 안 돌렸다. 단, v2의 광고 기본 임베더(BGE-small/base, GTE-large)도 전부
영어 중심이라 **한국어 약점은 버전이 아니라 구조적으로 공유**된다. 코퍼스는
QR 용량 때문에 450자로 잘랐고, 그 절단은 **양쪽에 동일 적용**했다. 이 실험은
"한국어 검색 품질" 한 축만 잰다 — memvid의 멀티모달·SDK 강점은 위에서 이미
인정했다.

## 한 줄

> memvid은 유통(16k★·Rust SDK·멀티모달)에서 앞서지만, 그 스타의 대부분은 이제
> 폐기된 비디오 기믹에서 왔고 세 헤드라인 숫자는 작성자 본인 글과 모순된다.
> 겹치는 유일한 축인 한국어 검색에서는 랜덤 수준(0.050 vs 우리 0.958)이다. Lemory는
> 유통에서 뒤지지만, 재현 가능한 정직한 실측과 한국어에서 이긴다 —
> 그리고 그 격차를 같은 하네스에서 숫자로 보여준다.

# 2026년 뜨는 "메모리" 챌린저들 — 같은 하네스 실측 (2026-07)

memvid 외에도 "AI 에이전트 메모리"를 표방하는 신생 프로젝트가 쏟아진다. 정찰로
12개를 추린 뒤, **로컬에서 실제로 돌릴 수 있는 것**은 같은 KorQuAD 하네스
(`benchmarks/run_challengers_korean.py`)에 올려 직접 쟀다. 돌릴 수 없는 것은
"주장 vs 구조"로 정직하게 서술하고 **실측하지 않았다고 명시**한다.

정찰의 핵심 관찰: **한국어를 튜닝한 것이 하나도 없고, 도구 간 동일-하네스
벤치를 공개한 것도 하나도 없다.** 그래서 그 두 축에서만 실측했다.

## 실측: 한국어 검색 (같은 코퍼스·질문, 전부 로컬)

KorQuAD 실문단 113개 · 인간 작성 질문 120개 · 문단 recall@1 · end-to-end 지연
(쿼리 임베딩·검색 전부 포함). 전부 API 없이 로컬.

| 시스템 | 아키텍처 | recall@1 | p50 지연 |
|---|---|---|---|
| **Lemory 키리스** | e5-small-ko-v2 + BM25(한글 바이그램) + 그래프 RRF | **0.967** | 21 ms |
| **Lemory fast** | 어휘 신호만 (임베딩 0회): 바이그램 BM25+오타교정+제목/시간 부스트 | **0.975** | **3.8 ms** |
| EchoVault v0.5 | SQLite FTS5 (오프라인 폴백) | 0.867 | 0.5 ms |
| Vestige v2.2.1 | nomic-embed-v1.5 벡터 + FSRS 인지검색 | 0.217 | 571 ms |

읽는 법 (냉정하게):

- **EchoVault의 FTS5는 진짜로 강하다 (0.867).** 블로우아웃 아니다. KorQuAD는
  질문이 문단과 어휘를 많이 공유해서 순수 FTS도 잘 맞는다 — 이걸 숨기지 않는다.
  그리고 FTS는 **압도적으로 빠르다(0.5ms).** Lemory가 이기는 10pt는 하이브리드
  (벡터가 어휘 안 겹치는 패러프레이즈 질문을 잡음) + 한글 바이그램 인덱싱에서
  온다. EchoVault의 벡터 경로는 Ollama가 필요해서 여기선 껐고(그렇게 라벨함),
  그 임베더(nomic)도 영어 중심이라 켜도 한국어에선 크게 안 오른다.
- **EchoVault의 강점(속도)은 이식했다: `mode=fast`.** 임베딩 없이 어휘 신호만
  쓰는 즉답 경로 — 같은 하네스에서 0.975 @ 3.8ms. KorQuAD처럼 질문이 문서
  어휘를 그대로 쓰는 코퍼스에선 fast가 하이브리드(0.967)마저 이긴다. 이것도
  숨기지 않는다: **하이브리드가 기본값인 이유는 패러프레이즈·크로스링구얼·
  멀티홉**(별도 가드 벤치 0.964/0.975/1.000)이며, 그 축들은 fast가 약하다.
  0.5ms인 EchoVault보다는 느린데(3.8ms), 그 차이는 오타 교정 + 바이그램 +
  시간/제목 부스트의 값이다. 검색창 as-you-type과 에이전트 폴링에는 fast,
  질문에는 hybrid — 둘 다 같은 인덱스라 전환 비용이 없다.
- **Vestige는 임베더를 켜고 쟀다 (0.217).** 키워드-only로 두면 자연어 질문에서
  영어·한국어 **양쪽 다 0.000**이라(조사·불용어 때문), 그건 불공정하다고 보고
  Vestige의 지연 임베딩(consolidate 사이클)을 돌려 벡터를 전부 생성한 **최선의
  구성**으로 쟀다. 그래도 0.217 — 기본 임베더 nomic-embed-v1.5가 **영어 우선**
  (memvid과 같은 계열)이라 한국어에서 무너진다. 멀티링구얼 옵션(qwen3)을
  스틸맨으로 시도했으나 v2.2.1에서 조용히 no-op(임베딩 0개 생성)이라 작동 안 함.
  게다가 인지 파이프라인 오버헤드로 **쿼리당 571ms** — Lemory의 27배.

## 먼저 인정할 진짜 강점 (실측 대상 아님, 그대로 인정)

- **EchoVault** (github mraza007/echovault): 포지셔닝이 가장 정직하고, 구조가
  단순하며(개발자 결정/이유/영향 스키마), FTS 지연이 사실상 0이다. "빠르고
  솔직한 개발자 메모리"로는 훌륭하다. Lemory엔 없는 미덕: 극단적 단순성.
- **Vestige** (github samvallad33/vestige): 진짜 신기한 인지 기능이 많다 —
  FSRS-6 간격반복 신뢰도, 모순 탐지(contradiction), dream/consolidation 재생,
  spreading-activation 그래프, 실패에서 원인을 거슬러 올리는 causal backfill,
  3D 시각화. "기억이 스스로 붕괴·강화·정리되는 인지 시스템"이 목표라면
  Vestige가 훨씬 앞선다. 우리가 이기는 건 오직 "한국어 검색 정확도·지연"
  한 축이다.

## 인정에서 끝내지 않고 가져온 것들

강점을 인정했으면 이식한다. 단, Lemory 방식으로 — LLM 0회, 기존 인덱스 재활용,
벤치마크에 영향 없는 opt-in:

| 챌린저의 강점 | Lemory 이식 | 구현 |
|---|---|---|
| EchoVault: sub-ms FTS 즉답 | `lemory search --fast` / `/search?mode=fast` | 어휘 신호만(바이그램 BM25·오타교정·제목/시간/사용 부스트), 임베딩 0회. 실측 0.975 @ 3.8ms |
| Vestige: 모순 탐지 | `lemory conflicts` / `/api/conflicts` | 기존 청크 벡터 행렬에서 노트 간 고코사인 쌍을 찾고 숫자 불일치·부정 충돌·중복을 분류. LLM 0회. `drift`가 "기억 vs 현실"이면 `conflicts`는 "기억 vs 기억" |
| Vestige: 검색 강도(자주 쓰는 기억 강화) | `usage_prior` 설정 (이미 있던 note_hits 활용) | fast 모드에도 적용 확장. 기본 0(벤치 정직성), 0.05-0.15 권장 |

이식하지 **않은** 것도 명시: FSRS-6 간격반복·dream 재생·3D 시각화는 노트 볼트
포지셔닝에 안 맞아 안 가져간다 (소스오브트루스는 사람이 쓴 .md — 시스템이
기억을 스스로 붕괴시키면 안 된다).

## 실측 안 한 것들 (정직하게: 안 쟀다)

돌리려면 각자 툴체인·모델 셋업이 필요해 이번엔 구조만 본다. **숫자로 이겼다고
주장하지 않는다.**

- **SwarmVault** (npm, KG 기반 옵시디언 대안): Karpathy식 위키 승인 워크플로가
  강점. 한국어 토크나이징·조사 처리 언급 없음 — 구조적으로 같은 갭 예상.
- **Engram** (Go 단일 바이너리, FTS5): 제로-디펜던시 Go 바이너리가 유통 강점.
  순수 FTS5라 EchoVault와 같은 계열 — 한국어에선 바이그램 없는 FTS의 한계 공유.
- **Vault Cortex** (npx, FTS5 + sqlite-vec + rerank): 아키텍처상 가장 강한
  챌린저. 리랭커가 있어 실측 가치가 높다. 기본 임베더가 영어권이면 같은 갭.
- **ai-memory-mcp**: "LongMemEval R@5 97%"(FTS5) 주장. 영어 벤치라 한국어와 별개.

## 한 줄

> 뜨는 챌린저들은 각자 진짜 강점이 있다 — EchoVault는 단순함과 속도, Vestige는
> 인지 기능(FSRS·모순·dream). 하지만 **한국어를 튜닝한 곳도, 도구 간 동일 하네스
> 벤치를 공개한 곳도 없다.** 그 두 빈칸이 Lemory의 자리다: 같은 코퍼스에서
> 한국어 recall@1 0.967(vs FTS 0.867, 인지-벡터 0.217)을, 재현 가능한 스크립트로
> 보여준다. 유통은 아직 뒤지지만, 정직한 실측과 한국어에서 이긴다.

# Tolaria (refactoringhq, 2026-04~) — AI 시대의 옵시디언

[Tolaria](https://github.com/refactoringhq/tolaria)는 Luca Rossi(Refactoring)의
마크다운 지식베이스 **데스크톱 앱**이다 (Tauri/React, AGPL). 1차 자료(README,
docs/ABSTRACTIONS.md, docs/ARCHITECTURE.md, AGENTS.md — 3,200줄)를 정독하고
판정한다.

## 카테고리가 다르다 — 그래서 비교 축을 정직하게 나눈다

Tolaria는 **편집기/관리자**다: 리치 에디터(BlockNote), 타입 시스템, Saved
Views, git 퍼스트 볼트, 그리고 앱 안에서 CLI 에이전트 8종(Claude Code, Codex,
Copilot, OpenCode, Pi, Antigravity, Kiro, Hermes)을 spawn해 MCP를 자동
배선하는 AI 워크스페이스. Lemory는 **메모리 미들웨어**다: 검색·기억·통합·
감사가 본업이고 편집기는 아예 없다.

## Tolaria가 이기는 축 (그대로 인정)

- **에이전트 온보딩 UX**: 앱을 깔면 에이전트 감지→MCP 배선→권한 모드
  (safe/power_user)까지 자동. 이건 진짜 잘 만들었다.
- **관리형 AGENTS.md**: 표준 가이던스 파일을 앱이 소유하고
  (managed/missing/broken/custom 상태 감지 + 복구), 사용자 작성 파일은
  불가침으로 존중한다.
- **편집기 그 자체**: 우리는 경쟁 자체를 안 한다 — Lemory는 옵시디언/아무
  편집기와 공존하는 미들웨어다.

## Lemory가 이기는 축 (1차 자료 근거)

- **검색이 카테고리가 다르다.** Tolaria의 검색은 자체 문서 기준
  "keyword-based search scans all vault .md files" — walkdir 스캔 + 인메모리
  키워드 인덱스가 전부다. 시맨틱도, 하이브리드 융합도, 오타 교정도, 시간
  인지도, 멀티홉 그래프 확장도 없다. 우리가 전 벤치에서 실측해온 그 축이
  통째로 비어 있다 (KorQuAD r@1 0.930 vs 키워드-only의 구조적 한계는 §5e
  Omnisearch 0.148에서 이미 측정된 계열).
- **기억 미들웨어 기능 부재**: 중복 통합, 승인 게이트, 드리프트/모순 감지,
  클라이언트 출처 표시, 되돌리기 타임라인, 세션 자동 기억 — 없음.
- **한국어**: 언급 자체가 없다.
- **벤치마크**: 검색 품질 측정을 공개하지 않는다 (편집기니까 당연하다 —
  그래서 "대비 좋은 게 없다"는 평은 검색·기억 축에선 성립하지 않는다).

## 흡수한 것 (이번 라운드 구현)

| Tolaria의 강점 | Lemory 이식 | 구현 |
|---|---|---|
| 관리형 AGENTS.md + 심 + 상태 감지/복구 | `lemory agents install` | 볼트 루트에 AGENTS.md(+CLAUDE.md/GEMINI.md 심)를 관리 마커와 함께 생성. managed/missing/broken/custom 판정, custom(사용자 작성)은 불가침, 깨진 관리 파일만 복구 |
| 에이전트 CLI 감지 + 셋업 | `lemory agents status` | claude/codex/gemini/copilot/opencode/cursor 감지(PATH + GUI 실행이 놓치는 표준 설치 경로) + 각각의 MCP 연결 한 줄 명령 출력 |
| git 퍼스트 (AutoGit 체크포인트) | `git_autocommit` 설정 (opt-in) | 볼트가 git 저장소면 AI 쓰기(save_memory/append)마다 자동 커밋 — 대시보드 되돌리기 위에 진짜 diff·revert가 얹힌다. 실패해도 쓰기를 막지 않는 best-effort |

이식하지 **않은** 것: 리치 에디터, 타입/Views/Neighborhood(편집기 UI —
콘솔 '지식' 탭의 로컬 그래프·관련 노트가 우리 몫의 대응), 앱 내 에이전트
spawn(우리는 미들웨어라 에이전트를 소유하지 않는다 — MCP로 연결될 뿐).

## 한 줄

> Tolaria는 "AI 시대의 옵시디언"이고, 잘 만든 편집기다. 하지만 검색은
> 키워드 스캔이 전부라, 기억·검색 축에서는 비교 자체가 성립하지 않는다.
> 우리가 배울 것은 에이전트 온보딩이었고 — AGENTS.md 관리, 에이전트 감지,
> git 체크포인트로 — 가져왔다. 이제 Lemory 볼트도 깔면 어떤 에이전트든
> 바로 기억으로 쓴다. 편집은 계속 옵시디언(또는 Tolaria!)에서 하면 된다 —
> 우리는 그 옆의 미들웨어다.
