## 문제

몇 년째 노트를 쌓아왔습니다. 데일리 로그, 회의록, 독서 메모, 반쯤 쓰다 만 아이디어 —
수천 개의 마크다운 파일. 그런데 정작 꺼낼 때는 `grep`이거나, 스크롤이거나, 포기입니다.
**대답하지 못하는 세컨드 브레인은 그냥 서류함입니다.**

업계의 답은 "전부 우리 클라우드에 올리세요, LLM으로 '메모리'를 추출해 드립니다,
호출당 과금입니다"입니다. 우리는 그게 거꾸로라고 생각합니다.

## 우리의 생각

1. **지식은 이미 당신의 볼트에 적혀 있고, 당신의 것입니다.** Lemory는 당신 컴퓨터에서,
   지금 그대로의 옵시디언 볼트 위에서 돕니다. SQLite 파일 하나. 도커도, 벡터DB도,
   누구의 클라우드도 없습니다. 폴더를 지우면 끝.
2. **모든 기억의 절반은 '언제'입니다.** 사람은 DB처럼 묻지 않습니다 —
   *"그거 뭐였지, 요새 내가 읽던 거?"* 처럼 묻습니다. 사실은 바뀌고, "지금 담당자"는
   최신 노트가 이겨야 하고, "지난주"는 키워드가 아니라 날짜 범위입니다.
3. **당신의 링크가 이미 지식 그래프입니다.** 남들은 LLM 비용을 태워 그래프를
   재구성하지만, `[[위키링크]]`는 당신이 직접 그린 엣지입니다. 멀티홉 질문
   (*"그 프로젝트 리드가 좋아하는 DB가 뭐더라?"*)은 여기서 풀립니다.
4. **주장에는 영수증이 필요합니다.** 아래 모든 숫자는 [`benchmarks/`](benchmarks/)로
   재현되고, 지는 항목도 그대로 공개합니다.

## 실데이터 증명 (자작 세트가 아니라)

**[KorQuAD 1.0](https://korquad.github.io/)** — 실제 한국어 위키피디아 140편,
사람이 쓴 질문 400개:

| | Recall@1 | Recall@5 | e2e 정답포함률 |
|---|---|---|---|
| **Lemory** | 0.908 | 0.985 | 0.900 |
| 벡터 RAG | 0.855 | 0.968 | 0.925 |
| BM25 | **0.923** | **0.993** | **0.950** |

<sub>**이 표는 BM25가 이깁니다. 그래도 싣습니다.** SQuAD류 질문은 문단을 보면서
작성돼 어휘가 그대로 겹치기 때문에 — 문서를 인용하는 질문엔 grep식 검색으로 충분합니다.
하지만 자기 기억을 인용문으로 검색하는 사람은 없죠. 같은 볼트에 **당신의 말로**
(패러프레이즈·한국어·오타) 물으면 BM25는 무너지고 Lemory는 버팁니다. 그게 아래 표고,
그게 이 제품입니다.</sub>

**나무위키 실문서 1,469편** (공개 덤프, 실제 위키링크 24,850개): full-support@8
**0.820 vs 0.660(벡터) vs 0.560(BM25)**, e2e F1 0.594 vs 0.562 vs 0.406.

**사람이 실제로 묻는 방식** (패러프레이즈/한국어 질문→영어 노트/키워드/오타):
Lemory **0.95~1.00** vs 벡터 0.46~0.54 vs BM25 0.25~0.58.

**타 솔루션 5종과 실측 비교** (동일 모델·코퍼스·저지): 멀티홉 **1.000** vs
mem0 0.579 / cognee 0.561 / supermemory 0.579 / LlamaIndex 0.649 / qmd 0.526 ·
LOCOMO 0.706(> mem0 발표치 0.669) · 검색 지연 ~3ms (타사 212ms~59초).

## 써보면 이런 느낌

적어둔 것이라면 뭐든 물어볼 수 있습니다 — 일, 공부, 게임, 생활:

```
$ lemory ask "3분기 킥오프에서 예산 얼마로 잡았지?"                # 회의
$ lemory ask "데이터플랫폼팀 리드가 누구고 무슨 일 하는 팀이지?"     # 조직
$ lemory ask "재택근무 정책, 작년이랑 지금이랑 뭐가 달라졌지?"       # 시점 간 정책 비교
$ lemory ask "자바스크립트 이벤트 루프 뭐였지? 내 노트 기준으로"     # 공부한 것
$ lemory ask "카오스 벨룸 가기 전에 준비물 뭐라고 적어놨더라?"       # 게임
$ lemory ask "알러지 올라올 때 대처 순서 뭐였지?"                   # 건강
$ lemory ask "전세 갱신 거절당하면 뭐부터 한다고 정리해놨지?"        # 법·행정
$ lemory ask "오사카에서 갔던 그 라멘집 이름이 뭐였지?"             # 여행
```

일반 RAG로는 구조적으로 안 되는 것들:

```
$ lemory ask "프로젝트 아틀라스 리드가 좋아하는 DB가 뭐더라?"
# 멀티홉: 아틀라스 노트 → [[리드]] 링크를 타고 → 그 사람 노트에서 답

$ lemory ask "요새 내가 읽던 책 뭐였지?"
요즘 읽는 책은 어스시의 마법사이다 [1, 3].    ← 넉 달 전 책이 아니라 지금 책
$ lemory ask "3월에 읽던 책은?"               ← 과거를 물으면 과거가 나옴

$ lemory recent    # 요새 뭐 만졌지? (LLM 없이 즉시)
$ lemory doctor    # 볼트/키/API/인덱스 원샷 진단
```

오타는 볼트 어휘로 자동 교정(API 0회), 목록 질문은 검색 폭 자동 확장.

## 시작하기 (2분)

```bash
pipx install "git+https://github.com/jwgo/lemory"
lemory setup      # 볼트 + 무료 Gemini 키 → 진단 + 첫 인덱싱
lemory ask "요새 내가 하던 그 프로젝트 어디까지 했지?"
lemory serve      # 웹 콘솔 + 실시간 볼트 감시
```

`lemory serve`는 검색창 하나가 아니라 **웹 콘솔**을 띄웁니다
(`127.0.0.1:8377`): **현황**(색인 통계·활동 피드·원클릭 재색인), **지식**(폴더
트리·태그·노트별 청크·백링크 탐색), **검색**(하이브리드/벡터/BM25 플레이그라운드
+ 출처 달린 답변), **설정**(검색 파라미터 즉시 적용, `lemory.toml`에 저장),
그리고 <kbd>⌘K</kbd> 팔레트로 어디든 점프.

## 100% 무료로 돌리기

**방법 1 — Gemini 무료 티어 (추천)**: [aistudio.google.com](https://aistudio.google.com)에서
2분이면 키가 나오고, 카드 등록이 없으면 과금될 방법 자체가 없습니다.
실제 소비량: 노트 1,000개 최초 색인 ≈ 임베딩 호출 47회(하루 한도의 5%),
내용이 안 바뀐 재색인은 **0회**(콘텐츠 해시 캐시), 질문은 **하루 약 250번** 무료.
한도에 걸리면 Lemory가 알아서 기다렸다 재시도합니다 — 요금이 아니라 대기입니다.

**방법 2 — 키 0개, 완전 오프라인**: `lemory setup`에서 번호만 고르세요.
② 완전 로컬(Ollama: **Gemma 3n E4B 4bit** 답변 생성 + **Qwen3-Embedding-0.6B**
검색, RAM 8GB+, 볼트가 컴퓨터 밖으로 안 나감) 또는 ③ 경량 검색 전용
(MiniLM 220MB, RAM 4GB — `ask`만 제외 전부 동작). 임베딩 모델을 바꾸면 벡터
공간이 달라지므로 **자동으로 전체 재색인**됩니다 (모델별 캐시라 되돌아가긴 공짜).

## 왜 잘하나 — 마법이 아니라 구조

- **멀티홉 1.000 vs 타사 0.53~0.58** — 당신이 그린 `[[위키링크]]`가 이미 지식
  그래프라서. 검색이 그 간선을 따라 1-hop 확장하되, 질의 유사도로 게이트하고
  직접 증거 아래로 캡을 걸어 노이즈는 못 올라옵니다. cognee는 같은 54노트
  볼트의 그래프를 만드는 데 LLM 호출로 45분을 쓰고도 점수가 낮았습니다 —
  Lemory의 그래프는 공짜입니다.
- **패러프레이즈/한국어 질문/오타에서 0.95+ vs 0.25~0.54** — 벡터와 BM25는
  서로 다른 방식으로 실패하므로 가중 RRF 융합이 양쪽을 커버. 한국어는 교착어라
  단어 단위 BM25가 깨지는데, 한글 바이그램 FTS로 해결. 모르는 단어는 검색 전에
  볼트 어휘로 교정(API 0회).
- **ms 단위 지연** — 벡터DB 서버 왕복이 없습니다. 실측: 청크 2천 개 5.5ms,
  1만 개 17ms, 5만 개 70ms. 실제 나무위키 3.3만 청크에선 질의당 약 0.2초 —
  cognee(5초)의 25배 속도.
- **비용은 0으로 수렴** — 임베딩은 내용 주소화 캐시라 같은 문단을 두 번
  임베딩하지 않고, compact 컨텍스트는 검색 증거를 ~550토큰 팩트시트로 압축.
- **시간 이해** — "지난주"는 날짜 범위, "요새"는 최신 우선, "3월에"는 과거 조회.

## 큰 볼트도 됩니다

벤치마크 최대 코퍼스는 실데이터입니다: **나무위키 1,469편, 청크 33,375개,
실제 위키링크 24,850개** — SQLite 파일 하나로 질의당 ~0.2초, recall@8 1.00.
색인은 증분식이라 노트 1만 개 볼트에서 오늘 5개를 고쳤으면 5개만 다시
처리합니다. 청크 2만 개를 넘으면 IVF-int8 인덱스로 자동 전환 — **청크 100만
개에서 5.9ms/질의, 정확 검색 대비 recall 1.000, RAM 4분의 1** (실측,
BENCHMARKS §12b). PDF 색인은 opt-in (`index_pdf = true`).

**옵시디언**은 플러그인 3파일 복사, **Claude Code/VS Code**는
`claude mcp add lemory -- lemory mcp` 한 줄 — 읽기 8툴 + **쓰기 2툴**
(save_memory·append_note): Claude가 결정·사실을 **볼트 안의 순수 마크다운
노트로** 저장합니다. 잠금 없음, 덮어쓰기 없음, 다음 질문부터 검색됩니다.
`tag:프로젝트 folder:회의록 예산` 같은 스코프 연산자, `lemory import-chats`
(ChatGPT/Claude 대화 내보내기 → 검색 가능한 노트)도 됩니다.
